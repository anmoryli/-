## 一、概述

### 1.1 文档目的

本文档详细描述孕期宝系统中**多源异构数据**从业务写入到向量检索的全链路处理方案，包括数据来源、处理策略、一致性保障及运维要点。

### 1.2 多源异构数据定义

系统存在 **5 类异构数据源**，最终汇聚到 Milvus 向量库供 AI 对话检索：

| 数据源 | 业务表 | 向量 source | 处理方式 |
|--------|--------|-------------|----------|
| 文字记录 | memo + text | memo | 标题+正文+URL |
| 语音记录 | memo + voice | memo | 标题+URL |
| 照片记录 | memo + photo | image_desc | 图像理解+用户描述+URL |
| 文件/PDF 记录 | memo + file | memo | PDFBox 抽全文 或 标题+URL |
| 对话消息 | message | message | 消息原文 |
| 产检 PDF | check_report | pdf | PDFBox 抽全文 |
| 知识库 | 不入 MySQL | 全局 user_id=-1 | Python 分块批量写入 |

### 1.3 架构全景

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          业务写入层                                       │
│  MemoController / MessageService / CheckReportService                    │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    │ 写入 MySQL + 提交嵌入任务
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                      embed_task 任务表                                   │
│  统一入口：可追溯、可重试、可审计                                          │
│  状态：pending → processing → success / failed                           │
│  动作：upsert（嵌入）| delete（删除）                                      │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    │ 定时扫描 pending（每 10 秒）
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                    EmbedTaskRunner 调度器                               │
│  1. 执行 upsert → RagService.embedSync → RAG /api/v1/embed              │
│  2. 执行 delete → RagService.deleteBySourceId → RAG /api/v1/delete       │
│  3. 失败重试（最多 3 次）                                                  │
│  4. 超限标记 failed 并打告警日志                                           │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                    Milvus 向量库 (yunji_rag)                             │
│  字段：id, vector, text, user_id, source, source_id, upload_time         │
│  按 source + source_id 定位单条，支持删除                                 │
└─────────────────────────────────────────────────────────────────────────┘
```

## 二、数据流转详解

### 2.1 新增/编辑记录 → 嵌入

| 步骤 | 执行方 | 说明 |
|------|--------|------|
| 1 | 业务层 | 写入 MySQL（memo/text/voice/photo/file） |
| 2 | 业务层 | 调用 `EmbedTaskService.submitUpsert(userId, text, source, sourceId)` |
| 3 | EmbedTaskService | 若同 source+sourceId 已有 pending 任务则更新快照，否则插入新任务 |
| 4 | EmbedTaskRunner | 每 10 秒扫描 pending，取 50 条批量处理 |
| 5 | EmbedTaskRunner | 调用 `RagService.embedSync` → RAG POST /api/v1/embed |
| 6 | RAG 服务 | 向量化后写入 Milvus |
| 7 | EmbedTaskRunner | 更新任务状态为 success |

### 2.2 删除记录 → 向量同步清理

| 步骤 | 执行方 | 说明 |
|------|--------|------|
| 1 | MemoServiceImpl.deleteMemo | 级联删除 MySQL（memo + text/voice/photo/file） |
| 2 | MemoServiceImpl | 删除 OSS 资源 |
| 3 | MemoServiceImpl | 调用 `embedTaskService.submitDelete(userId, "memo", memoId)` |
| 4 | MemoServiceImpl | 若为照片记录，额外 `submitDelete(userId, "image_desc", memoId)` |
| 5 | EmbedTaskRunner | 扫描到 delete 任务，调用 `RagService.deleteBySourceId` |
| 6 | RAG 服务 | DELETE /api/v1/delete?source=xxx&source_id=xxx |
| 7 | Milvus | 按 filter 表达式删除对应向量 |

### 2.3 各数据源接入点

| 数据源 | 接入位置 | 调用方式 |
|--------|----------|----------|
| 文字记录 | MemoController.addText | submitUpsert(标题+正文) |
| 文字记录编辑 | MemoController.updateText | submitUpsert(标题+正文) |
| 语音记录 | MemoController.addVoice | submitUpsert(标题+[URL]) |
| 照片记录 | MemoController.addPhoto | CompletableFuture 内：图像理解后 submitUpsert |
| 文件记录 | MemoController.addFile | 非 PDF：submitUpsert；PDF：CompletableFuture 内抽全文后 submitUpsert |
| 对话消息 | MessageServiceServiceImpl.save | submitUpsert(content) |
| 产检 PDF | CheckReportServiceImpl.uploadAndParse | submitUpsert(extractedPdfText) |
| 记录删除 | MemoServiceImpl.deleteMemo | submitDelete(memo) + submitDelete(image_desc)（照片时） |

## 三、核心组件说明

### 3.1 embed_task 表结构

```sql
CREATE TABLE embed_task (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    source VARCHAR(20) NOT NULL,        -- memo|message|pdf|image_desc
    source_id VARCHAR(50) NOT NULL,    -- 业务表主键
    action VARCHAR(10) NOT NULL DEFAULT 'upsert',  -- upsert|delete
    text_snapshot TEXT NULL,           -- 嵌入文本快照，delete 时为空
    status VARCHAR(20) NOT NULL DEFAULT 'pending', -- pending|processing|success|failed
    retry_count INT NOT NULL DEFAULT 0,
    max_retry INT NOT NULL DEFAULT 3,
    error_msg VARCHAR(500) NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_status_created (status, created_at),
    INDEX idx_source (source, source_id),
    INDEX idx_user (user_id)
);
```

### 3.2 状态机

```
创建/编辑记录                    删除记录
      │                              │
      ▼                              ▼
  [pending]                     [pending] (action=delete)
      │                              │
  EmbedTaskRunner 扫描               │
      │                              │
      ▼                              ▼
  [processing]                 [processing]
      │                              │
  成功    失败                      成功
   │       │                         │
   ▼       ▼                         ▼
[success] retry_count++          [success]
            │
        < 3次 ? 
      是 │     │ 否
         ▼     ▼
    [pending] [failed] → 告警
```

### 3.3 RAG 服务接口

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/v1/embed | POST | 单条文本向量化写入，body: user_id, text, source, source_id |
| /api/v1/delete | DELETE | 按 source+source_id 删除向量，query: source, source_id |

### 3.4 Java 端 RagService 方法

| 方法 | 说明 |
|------|------|
| embedAsync | 异步嵌入（保留兼容，新逻辑已迁移至 embed_task） |
| embedSync | 同步嵌入，供 EmbedTaskRunner 使用，异常向上抛出 |
| deleteBySourceId | 按 source+source_id 调用 RAG 删除接口 |

## 四、已解决问题清单

| # | 问题 | 解决方案 |
|---|------|----------|
| 1 | 删除记录时向量库未同步清理 | MemoServiceImpl.deleteMemo 末尾 submitDelete，调度器执行 RAG delete |
| 2 | 嵌入失败静默丢失 | embed_task 记录失败，重试 3 次，超限标记 failed 并打日志 |
| 3 | 编辑记录后向量库仍是旧内容 | 编辑时 submitUpsert 会更新同 sourceId 的 pending 任务快照，或新建任务覆盖 |
| 4 | 无法审计嵌入状态 | embed_task 表可查询 status、retry_count、error_msg |
| 5 | 异步时序竞争（AI 标题 vs 嵌入） | 可扩展：引入 waiting_enrich 状态，AI 丰富化完成后 activateTask |

## 五、运维与监控

### 5.1 关键日志

- `[EmbedTask] 提交嵌入任务 source=xxx sourceId=xxx`
- `[EmbedTask] 提交删除任务 source=xxx sourceId=xxx`
- `[嵌入] 任务最终失败 taskId=xxx source=xxx sourceId=xxx`
- `[嵌入] 任务重试 1/3 taskId=xxx`

### 5.2 监控建议

1. **失败任务告警**：定时查询 `status='failed'` 且 `updated_at` 在最近 1 小时内，数量 > 0 时告警
2. **堆积监控**：`status='pending'` 数量持续增长需排查 RAG 服务或网络
3. **对账任务**（可选）：每日凌晨比对 MySQL 与 embed_task 成功记录，补建缺失、清理脏数据

### 5.3 数据库迁移

执行 `migration_v39_embed_task.sql` 创建 embed_task 表。若使用 db-all.sql 全量建库，已包含 embed_task 表定义。

## 六、扩展与待办

### 6.0 外部数据扩展（规划）

面向 API/社区/论文等外部搜集数据的扩展路线、表结构、调度与合规要求，见 **[外部数据扩展技术方案](./外部数据扩展技术方案.md)**。

### 6.1 统一嵌入文本格式（可选）

可引入 `EmbedTextFormatter`，将各来源文本统一为结构化模板，提升检索精度：

```
[来源:memo] [类型:文字] [孕周:12周] [日期:2026-03-15] [心情:happy]
标题：第一次产检
正文：……
[URL] https://xxx
```

### 6.2 检索结果回查（可选）

在 AI 拼接 context 前，对 RAG 返回的每条 hit 回查 MySQL 确认记录存在且当前用户有权限，过滤已删除/无权限结果。

### 6.3 数据对账任务（可选）

每日定时任务：MySQL 有但向量库缺失 → 补建 embed_task；向量库有但 MySQL 已删 → 提交 delete 任务；输出对账报告。
