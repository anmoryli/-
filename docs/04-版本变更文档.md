# 孕期宝 版本变更文档

本文档基于迭代过程中的需求与问题修复整理，按功能域与时间顺序记录主要变更。

---

## 一、时间轴与记录

### 1.1 时间轴展示与标签

- **问题**：时间轴页面同一条记录被渲染两次；AI 生成的标签过长，超出展示预期。
- **变更**：
  - 去除重复的「心情 + 分类」渲染块，避免一条记录展示两遍。
  - 标签展示做长度限制：超过 6 字时截断并加省略号（如 `xxx…`），保证界面统一且完整显示 6 字内标签。

### 1.2 记录与向量库

- **需求**：新文字记录需写入向量库；PDF 上传需抽取正文并写入向量库；图片记录需结合图像理解结果再写入向量库，且带 URL。
- **变更**：
  - 文字/语音记录：保存后异步调用 RAG 服务的 `/api/v1/embed`，写入内容并带 `[URL]`（若存在）。
  - PDF 上传：后台异步用 PDFBox 抽取正文，抽取结果与文件 URL 一并写入向量库；非 PDF 文件用标题 + `[URL]` 嵌入。
  - 图片记录：先调用图像理解模型，将「用户描述 + 图像理解结果」拼接为固定句式，并追加 `[URL]` 后异步嵌入；嵌入与“保存成功”提示异步执行，不阻塞返回。

### 1.3 检索结果中的 URL 与前端展示

- **需求**：嵌入时包含 URL；检索结果若含 URL，AI 回复中需保留；前端对图片 URL 渲染为图片，文件 URL 渲染为可点击链接。
- **变更**：
  - 嵌入侧：文件/图片类记录在嵌入文本中统一加入 `[URL] xxx`。
  - 对话侧：在 system 中明确要求参考内容里的图片用 `![描述](url)`、文件用 `[点击查看](url)`；必要时从 RAG 返回文本中解析图片 URL 并追加到回复末尾，保证前端能拿到图片地址。
  - 前端：Markdown 渲染组件对 RAG 风格 `[URL] https://...` 及裸图片 URL 做预处理，转为 `![图片](url)`；`img`/`a` 使用 `referrerPolicy="no-referrer"` 以改善外链图片加载。

## 二、AI 对话与意图

### 2.1 多模态意图识别

- **问题**：“找图”与“生成图”被混淆，导致本应检索的请求走了图生图。
- **变更**：
  - 在意图识别 prompt 与代码中明确：仅当用户明确表达「生成图/画图/改图/风格迁移/变体」时才判为 TEXT_TO_IMAGE/IMAGE_TO_IMAGE；「找图/搜图/查图/查找图片」等判为 TEXT_CHAT 或 IMAGE_UNDERSTANDING，不判为图生图。
  - 若模型仍将“查找”类判为 *_TO_IMAGE，则代码中按关键词（找/搜/查 + 无生成类词）强制改为 TEXT_CHAT 或 IMAGE_UNDERSTANDING。

### 2.2 RAG 检索与不嵌入

- **需求**：从向量库“找东西”时，不要把用户问题再写入向量库。
- **变更**：
  - `MessageService` 增加 `save(..., embedInRag)`；当 `embedInRag == false` 且为用户消息时不调用 `ragService.embedAsync`。
  - 流式主路径（会执行 RAG 检索）在调用 RAG 前先保存用户消息，并传入 `embedInRag=false`；提醒分支保存用户消息时传入 `embedInRag=true`；非流式 `/chat` 未走 RAG，保持默认嵌入。

### 2.3 RAG 检索与日志

- **需求**：后台能清晰看到 RAG 调用与结果，便于排查。
- **变更**：
  - 在调用 RAG 前打 `[RAG] 开始检索`；检索完成后打结果长度与结果内容概要；检索失败时用 WARN 级别并带异常信息。
  - RAG 服务内对请求与响应做必要日志，便于端到端排查。

### 2.4 流式与前端超时

- **问题**：AI 不流式输出或前端发送按钮一直不可用。
- **变更**：
  - 前端 `chatStream` 使用 `AbortController` 做 2 分钟超时，超时后中止请求并恢复发送按钮状态；失败时 `toast.error` 提示。
  - 发送按钮禁用条件明确包含 `!user`，在用户未加载或未授权时禁用。
  - Next.js 代理层：使用 undici 自定义 Agent，将 headersTimeout 设为 10 分钟、bodyTimeout 设为 0，避免代理在等待首包时过早超时；FormData 按字段重建后转发，避免参数丢失。

### 2.5 userId 与会话

- **问题**：流式接口有时未传 `userId`，导致参数校验异常。
- **变更**：
  - `chat-stream` 的 `userId` 改为可选；若未传则通过 `conversationId` 查会话得到 `userId`；若仍无法解析则返回错误并结束，避免 500。
  - 在 lambda 中使用 `userId` 时，保证其为 effectively final（通过中间变量解析后赋给 final 变量）。

### 2.6 图片 URL 正则与 RAG 解析

- **问题**：从 RAG 文本中提取图片 URL 时使用的正则存在“Unclosed group”，导致 PatternSyntaxException。
- **变更**：
  - 将复杂单正则拆成两个简单正则：一是 `https?://.../photo/...`；二是以常见图片扩展名结尾的 `https?://...`，避免括号未闭合错误，同时保留对 OSS 图片路径的识别。

## 三、定时提醒（AI 添加任务）

### 3.1 多轮保存与校验

- **问题**：多轮对话收集的提醒有时未写入数据库。
- **变更**：
  - 对 `newTime` 等时间字符串做更稳健的解析（含“明天下午 13:00”等），解析失败时回退到 `schedule_parse` 提示词用 LLM 解析。
  - 插入 `scheduled_operation` 后增加存在性校验（如 `selectById`），仅当库中能查到该条时才向用户返回成功并附带表格；否则提示“可能未成功保存，请重试”。

### 3.2 规则与轮次

- **问题**：提醒被误判为“每天”；未收集齐时间/地点/做什么就创建；轮次过多且无表格反馈。
- **变更**：
  - **schedule_parse**：默认 `scheduleType=once`，仅当用户明确说「每天/天天/每日」且给时间时才设为 daily。
  - **schedule_task_collect**：重写 prompt，要求最多 2 轮、单轮尽量抽全、第二轮必须合并已有状态并 `complete:true`；缺项时用默认值（如“明天 09:00”“待定”“待办”）仍写入，避免卡死。
  - 多轮状态用 `ScheduleDraftStore`（内存）维护；收集时注入近期对话摘要（`conversationSnippet`），便于模型理解上下文。
  - 成功写入并校验通过后，在回复中追加 Markdown 表格：时间、地点、做什么、备注、类型（once/daily），时间列使用 `nextRunAt` 格式化为 `yyyy-MM-dd HH:mm`。

### 3.3 触发时的文案改写

- **需求**：提醒触发时下发的文案要“带入当前场景”（如“明天去人民公园” → “准备出发去人民公园吧”）。
- **变更**：
  - 新增 `ReminderRewriteService`，在 `ScheduledOperationRunner` 发送站内通知/邮件前，先调用 LLM（`reminder_send_rewrite` prompt）对原始内容改写，再将改写结果作为通知正文与邮件正文。

## 四、家庭与通知

### 4.1 “添加配偶”类通知仅一次

- **问题**：用户未添加配偶时，站内消息反复推送“添加配偶”等提示。
- **变更**：
  - 在插入此类通知前，按 `user_id + type + title` 查询是否已有记录（如 `countByUserIdAndTypeAndTitle`）；若已有则不再插入，保证同一用户同类型同标题只提醒一次。
  - 涉及逻辑分布在配偶提及邮件、邀请配偶加入、添加配偶后可分配任务等通知处。

### 4.2 配偶提及与邮件

- **逻辑**：对话中检测到@配偶时，向配偶邮箱发送摘要邮件；若为配偶本人发起的对话则向孕妇（家庭创建者）发邮件。邮件发送失败仅记录日志，不阻塞对话完成与消息保存。

## 五、情绪价值与 AI 能力拓展

### 5.1 心情与共情

- **变更**：在 `ai_chat_system` 及非流式/配偶版 system 中增加“情绪与共情”规则：当用户表达疲惫、焦虑、开心等时，先简短共情再给建议或记录灵感；涉及累/烦/担心/开心/紧张等词时优先情感支持。

### 5.2 写信给宝宝

- **变更**：在 `pregnancy.json` 中新增 `letter_to_baby_guide` prompt（结构、润色、结合记录）；当用户问题包含“写信”且“宝宝/给宝宝/对宝宝”时，在 system 中追加该引导；前端快捷问题增加「帮我想一句对宝宝说的话」。

### 5.3 每日暖心语

- **变更**：新增 `daily_warm_message` prompt（按孕周与日期生成一句鼓励）；后端新增 `GET /api/ai/daily-warm?userId=xx`，返回一句暖心语，失败时返回默认句；前端在聊天页无消息时请求并展示该句（斜体样式）。

### 5.4 配偶端支持

- **变更**：在 `ai_chat_system_spouse` 中增加情绪支持与沟通话术：如何安慰妻子、倾听、表达关心、小惊喜建议（如录语音、写一句对宝宝的话、陪记录）；继续强调医疗问题建议咨询医生。

- **变更**：日期分组块增加 `animate-in fade-in duration-300`，与整体动效统一。

## 七、基础设施与稳定性

### 7.1 Redis 缓存

- **问题**：Redis 写失败（如 RDB 持久化失败导致 MISCONF）时，若直接抛异常会导致请求失败。
- **变更**：
  - 实现 `CachingConfigurer` 与自定义 `CacheErrorHandler`：对 Get/Put/Evict/Clear 错误仅打 WARN，不重新抛出异常，业务自动回退到查库，保证 Redis 不可用时系统仍可长期运行。

### 7.2 MySQL 与定时任务

- **问题**：定时任务启动或执行时若遇 MySQL 连接/读超时，会拖垮启动或持续报错。
- **变更**：
  - 数据源与 HikariCP：JDBC URL 增加 `connectTimeout=15000&socketTimeout=30000`；`connection-timeout=20000`，`initialization-fail-timeout=-1`，避免库暂时不可用时直接启动失败。
  - `ScheduledOperationRunner.processDue`：对 `selectDue` 等数据库调用做 try-catch，发生连接/超时类异常时打 WARN 并跳过本轮，下次调度再试，不中断应用。

## 八、小结

上述变更覆盖：时间轴与标签展示、记录与向量库写入（含 PDF/图片）、RAG 检索与 URL 展示、意图识别与流式/超时/userId、定时提醒多轮与校验与改写、家庭与通知去重、情绪价值与每日暖心语与配偶支持、前端动效与 API、Redis/MySQL 容错等。版本变更文档可按发布周期从本节中摘录形成发布说明。
