# 孕期宝多源异构数据处理改进建议（外部数据扩展）

> 基于现有 v2.0 架构，针对“外部搜集的数据”提出增强方案  
> 创建时间：2026-03-20  
> 状态：建议方案，待评审

---

## 一、当前系统已覆盖的数据源

现有系统已处理 5 类异构数据，通过 embed_task 任务表统一管理向量嵌入：

| 数据源 | 向量source | 处理方式 | 备注 |
|--------|------------|----------|------|
| 文字/语音/照片/文件记录 | memo, image_desc | 业务写入+embed_task异步嵌入 | 用户产生数据 |
| 对话消息 | message | 消息原文嵌入 | 用户产生数据 |
| 产检PDF | pdf | PDFBox抽全文嵌入 | 用户产生数据 |
| **知识库文档** | 全局user_id=-1 | Python分块批量写入 | **已有外部数据** |

**当前外部数据（知识库）的局限**：
1. **来源单一**：主要是静态PDF/DOCX/MD文件
2. **更新频率低**：批量导入，非实时更新
3. **结构化程度有限**：文档抽取后转为纯文本，丢失表格、图表等结构信息

## 二、针对“外部搜集的数据”的改进方向

### 2.1 扩展外部数据源类型
在现有 `source` 枚举基础上新增类别，适配更多外部数据：

| 新增source | 描述 | 示例 | 优先级 |
|------------|------|------|--------|
| `medical_api` | 医疗健康API数据 | 症状库、药品库、检查指标参考范围 | 高 |
| `community_post` | 社区/论坛数据 | 孕妇讨论帖、经验分享 | 中 |
| `research_paper` | 学术论文 | PubMed/知网论文摘要 | 中 |
| `device_data` | 穿戴设备数据 | 胎心监测、运动步数（需脱敏聚合） | 低 |
| `third_party_record` | 第三方健康记录 | 医院HIS系统对接（需授权） | 低 |

**实现路径**：
- 扩展 `embed_task.source` 枚举长度（VARCHAR(50) → VARCHAR(100)）
- 为每类外部数据开发对应的采集器（crawler/API client）
- 采集器将原始数据转为文本后调用 `EmbedTaskService.submitUpsert`

### 2.2 外部数据质量与可信度管理
当前系统缺乏数据质量评估，对外部数据尤为重要：

```sql
-- 扩展现有 embed_task 表（可选字段）
ALTER TABLE embed_task ADD COLUMN (
    data_source VARCHAR(100) COMMENT '数据来源标识，如"api.symptom_library"',
    confidence_score DECIMAL(3,2) DEFAULT 1.0 COMMENT '可信度评分0-1',
    freshness_days INT COMMENT '数据新鲜度（采集到当前天数）',
    license_info VARCHAR(200) COMMENT '使用许可信息'
);
```

**质量评估规则**：
- **权威性**：三甲医院指南（1.0）vs 个人博客（0.6）
- **时效性**：2026年指南 vs 2010年文献（影响检索排序权重）
- **合规性**：自动标记无明确 license 的外部数据

### 2.3 实时外部数据流接入
当前 embed_task 是定时扫描（每10秒），适合业务数据。外部数据可能需要更实时：

**方案A：优先队列**（推荐）
```java
// 在 EmbedTaskRunner 中区分优先级
@Scheduled(fixedDelay = 2000)  // 2秒扫描一次
public void processHighPriorityTasks() {
    List<EmbedTask> highPriority = embedTaskMapper.selectPendingBySourceTypes(
        Arrays.asList("medical_api", "device_data"), 
        Limit.of(50)
    );
    // 优先处理
}
```

**方案B：直接调用RAG接口**
对时效性要求极高的数据（如实时胎心异常），绕过 embed_task，直接调用 `RagService.embedSync`，但需记录日志以便审计。

### 2.4 结构化外部数据增强检索
外部数据往往包含结构化信息（药品剂量、检查指标范围），纯文本嵌入会丢失这些信息：

**改进方法**：
```python
# 在 RAG 的 embed 接口前增加结构化提取
def enrich_structured_context(text, source):
    if source == "medical_api":
        # 提取指标、单位、参考范围
        extracted = extract_medical_entities(text)
        return f"""
        指标: {extracted['metric']}
        正常范围: {extracted['normal_range']}
        单位: {extracted['unit']}
        说明: {text}
        """
    return text
```

### 2.5 外部数据权限与隔离
当前知识库使用 `user_id=-1` 表示全局数据。但部分外部数据可能需要更细粒度控制：

- `user_id=-1`：完全公开（医学常识）
- `user_id=0`：机构共享（合作医院指南）
- `user_id<0` 的其他负值：数据源分组（方便批量下线）

### 2.6 外部数据更新与淘汰机制
避免向量库积累过期信息：

## 三、具体实施优先级建议

### 3.1 短期（1-2周）
1. **评估新增数据源**：与业务方确定最急需的2-3类外部数据
2. **扩展 source 枚举**：修改表结构，增加 `medical_api`、`community_post` 等
3. **开发首个外部采集器**：如症状库API对接，验证全链路

### 3.2 中期（1个月）
1. **实现质量评分**：在 embed_task 增加 confidence_score，影响检索权重
2. **结构化提取**：针对医疗数据优化文本格式化模板
3. **权限细化**：区分公开数据与机构共享数据

### 3.3 长期（季度）
1. **实时流支持**：高优先级队列或直接同步嵌入
2. **自动更新机制**：外部数据定期刷新
3. **数据血缘可视化**：追溯每个向量片段的来源、采集时间、质量评分

## 四、注意事项与风险控制

### 4.1 合规性要求
- **数据安全法**：确保采集“合法、正当、必要”
- **用户知情同意**：涉及用户个人健康数据的需明确授权
- **第三方协议**：API数据需遵守提供方使用条款

### 4.2 性能影响
- **向量库规模**：新增数据源会增加 Milvus 内存与查询延迟
- **监控指标**：增加以下监控项：
  - 外部数据占向量库比例
  - 外部数据查询命中率
  - 外部数据更新延迟

### 4.3 测试策略
- **沙箱环境**：外部数据先在测试环境全链路验证
- **数据隔离**：生产向量库需能按数据源隔离测试数据
- **回滚机制**：支持按 source 批量下线问题数据
