## 一、系统架构

### 1.1 整体架构

采用 **前后端分离 + 微服务解耦** 的四层架构：

```
┌──────────────────────────────────────────────┐
│              客户端层                          │
│   Next.js 前端  │  Flask 管理后台  │  浏览器    │
└──────────────────────────────────────────────┘
                     │
┌──────────────────────────────────────────────┐
│            接入层 / API 网关                    │
│        Next.js API 路由（BFF 代理层）           │
│   undici 自定义 Agent，headersTimeout=10min    │
│   bodyTimeout=0，FormData 按字段重建转发        │
└──────────────────────────────────────────────┘
                     │
┌──────────────────────────────────────────────┐
│             后端服务层                         │
│        Spring Boot 3.2.2 (Java 17+)          │
│   REST API / SSE 流式 / MyBatis / AOP         │
│   Spring Cache / 定时调度 / 异步任务            │
└──────────────────────────────────────────────┘
                     │
┌──────────────────────────────────────────────┐
│           数据与 AI 层                         │
│  MySQL 8  │  Redis  │  RAG 服务(Python+Milvus) │
│  阿里云 OSS │  大模型 API (OpenAI 兼容)         │
└──────────────────────────────────────────────┘
```

### 1.2 服务间通信

| 通信路径 | 协议 | 说明 |
|----------|------|------|
| 前端 → Next.js API 路由 | HTTP/SSE | BFF 代理，统一转发至后端 |
| Next.js API 路由 → Spring Boot | HTTP/SSE | undici 自定义超时，避免代理缓冲中断流式 |
| Spring Boot → RAG 服务 | HTTP REST | OkHttp 客户端，连接超时30s，读超时120s |
| Spring Boot → 大模型 API | HTTP/SSE | Spring AI OpenAI Starter + OkHttp 原生调用 |
| Spring Boot → MySQL | JDBC | HikariCP 连接池，connectTimeout=15s，socketTimeout=30s |
| Spring Boot → Redis | Spring Data Redis | Lettuce 客户端，容错降级 |
| Spring Boot → 阿里云 OSS | SDK | 阿里云 OSS Java SDK 3.18.0 |

### 2.7 前端关键技术实现

1. **SSE 流式消费**：使用 `AbortController` + 2 分钟超时控制；逐块接收 AI 回复并追加到消息气泡；超时自动中止并恢复发送按钮
2. **BFF 代理层**：Next.js API 路由使用 undici 自定义 Agent（headersTimeout=10min, bodyTimeout=0），FormData 按字段重建转发，避免流式代理超时与参数丢失
3. **Markdown 预处理**：对 RAG 风格 `[URL] https://...` 及裸图片 URL 自动转为 `![图片](url)`；`<img>`/`<a>` 使用 `referrerPolicy="no-referrer"` 改善外链加载
4. **动效系统**：消息气泡 fade-in/slide-in 动画；流式回复尾部脉冲光标；日期分组块 `animate-in fade-in duration-300`
5. **每日暖心语**：聊天页无消息时请求 `GET /api/ai/daily-warm`，斜体展示；有消息后隐藏

### 3.4 Spring AI 集成实现

```java
// SpringAiConfig.java - ChatClient 配置
@Bean
public ChatClient chatClient(OpenAiChatModel openAiChatModel) {
    return ChatClient.create(openAiChatModel);
}

// 使用方式：所有 AI 调用统一通过 ChatClient.builder(openAiChatModel)
ChatClient.builder(openAiChatModel)
    .defaultSystem(systemPrompt)
    .build()
    .prompt()
    .user(userPrompt)
    .call()
    .content();
```

- 配置基于 `spring.ai.openai.*`，支持 OpenAI 兼容接口
- 图像理解使用 OkHttp 原生调用（需发送 multimodal content array）
- 模型名配置为 `gpt-5.2`（通过 OpenAI 兼容代理指向实际模型）

### 3.5 提示词管理系统

采用 **数据库优先 + JSON 兜底** 的双层提示词管理：

1. **数据库层**：`prompt_template` 表，按 `key + model_type` 存储，支持运行时热更新
2. **JSON 层**：`resources/prompts/pregnancy.json`，启动时加载，作为未入库提示词的兜底
3. **查询顺序**：先查数据库 → 未命中则查 JSON → 支持 `{placeholder}` 模板变量替换

### 4.2 Redis 缓存

#### 缓存架构

```
业务服务 ──@Cacheable──▶ Redis ──miss──▶ MySQL
                         ↓ (错误)
                    仅打 WARN 日志
                    自动降级到 MySQL
```

#### 配置细节

- **CacheManager**：`RedisCacheManager`，Key 使用 `StringRedisSerializer`，Value 使用 JSON 序列化
- **默认 TTL**：10 分钟
- **命名空间**：
  - `user`：用户信息缓存，TTL 10 分钟，使用专用 User 序列化器（不写 `@class`，兼容旧数据）
  - `familyMembers`：家庭成员列表，TTL 5 分钟
- **容错机制**：实现 `CachingConfigurer` + 自定义 `CacheErrorHandler`，对 Get/Put/Evict/Clear 四类 Redis 操作错误**仅打 WARN 日志，不抛异常**
- **设计理念**：Redis 作为加速层，非强依赖；MISCONF（RDB 持久化失败）等场景下业务零中断

### 4.3 Milvus 向量数据库

- **集合**：`yunji_rag`（用户私有 + 百科统一存储）
- **字段**：`id`（主键）、`vector`（1024 维 float）、`text`（原始文本）、`user_id`（多租户隔离）、`source`（来源类型）、`source_id`（来源ID）、`upload_time`
- **索引**：向量字段 AUTOINDEX，内积（IP）相似度
- **多租户**：`user_id > 0` 为用户私有数据，`user_id = -1` 为全局百科
- **写入方**：仅 Python RAG 服务执行写入；Java 通过 HTTP 调用 `/api/v1/embed` 提交

### 4.4 阿里云 OSS 对象存储

- **存储内容**：图片、语音、视频、PDF、头像等多媒体文件
- **上传限制**：单文件最大 500MB
- **路径规则**：按 userId 分目录组织
- **功能**：支持 PDF 内嵌图片上传（扫描件 OCR 场景）

## 五、AI 与 RAG 技术详解（核心亮点）

### 5.1 RAG 服务架构

独立的 Python FastAPI 服务，与 Java 后端通过 HTTP 解耦：

| 接口 | 方法 | 功能 |
|------|------|------|
| `/api/v1/query` | GET | 向量检索，支持 user_id/include_global/top_k 参数 |
| `/api/v1/embed` | POST | 文本向量化写入，接收 user_id/text/source/source_id |
| `/api/v1/upload` | POST | 一键重建知识库（百科） |

#### Java 端 RagService 实现

```java
// 检索：OkHttp GET 调用，解析 JSON 响应中的 hits 数组
public String getRelevant(String question, Integer userId, boolean includeGlobal, int topK)

// 嵌入：@Async 异步写入，失败仅打日志不阻塞业务
@Async
public void embedAsync(int userId, String text, String source, String sourceId)
```

- 连接超时：30 秒
- 读取超时：120 秒
- 文本截断：最大 10000 字
- 默认 top_k：10（最大 20）

### 5.2 多源异构分块策略

| 数据来源 | 分块策略 | 分块参数 | 执行位置 | 设计理由 |
|----------|----------|----------|----------|----------|
| 知识库 PDF/DOCX/MD | **分块** | chunk_size=1500字, overlap=100字 | Python RAG 服务 | 长文档整篇嵌入时单向量难以细粒度匹配 |
| 用户记录 memo | **整条** | 截断上限 10000 字 | Java → RAG | 单条记录是独立语义单元 |
| 对话消息 message | **整条** | 截断上限 10000 字 | Java → RAG | 保留对话上下文完整性 |
| 产检 PDF | **整条** | PDFBox 抽全文后截断 | Java PDFBox → RAG | 一份报告一个语义单元 |
| 图片描述 | **整条** | 描述+理解结果+URL | Java → RAG | 短文本完整保留 |

#### 知识库分块细节

- **加载器**：PDF 用 `PyPDFLoader` 按页加载；DOCX 用 python-docx 按段落拼接；MD 直接读取
- **分块器**：`RecursiveCharacterTextSplitter`
- **分隔符优先级**：`["\n\n", "\n", "。", "！", "？", "；", " ", ""]`
- **清洗**：`clean_text()` 去除页眉页脚（`—1—`、`第 N 页`）、多余空行、纯数字行、`\r`
- **嵌入**：每个 chunk 独立调用 Embedding API 生成 1024 维向量

### 5.3 AI 对话完整流程

```
用户发送消息
    │
    ▼
[意图识别] ChatClient + 意图识别 prompt
    │
    ├──▶ REMINDER → 多轮收集（最多2轮）→ 写入 scheduled_operation → 回复+表格
    ├──▶ TEXT_TO_IMAGE → 调用图像生成 API → 返回生成图
    ├──▶ IMAGE_TO_IMAGE → 调用图像生成 API → 返回生成图
    ├──▶ IMAGE_UNDERSTANDING → 图像理解 → 拼接描述 → 走 RAG 对话
    └──▶ TEXT_CHAT → RAG 检索 → system prompt 拼接 → 大模型流式生成
                │
                ├── 保存用户消息（embedInRag=false，检索轮不嵌入）
                ├── RAG 检索 top_k=10
                ├── 拼 system prompt + 上下文 + 历史消息
                ├── 流式生成（SSE）
                ├── 保存 AI 回复
                ├── 配偶提及检测 → 邮件通知
                └── 从 RAG 结果解析图片 URL → 追加到回复末尾
```

#### 意图识别规则

- **TEXT_TO_IMAGE/IMAGE_TO_IMAGE**：仅当用户明确说「生成图/画图/改图/风格迁移/变体」
- **TEXT_CHAT**：「找图/搜图/查图/查找图片」强制判为 TEXT_CHAT（不走图生图）
- **IMAGE_UNDERSTANDING**：有图且未要求生成时
- **REMINDER**：涉及时间+提醒意图时

#### 检索不嵌入机制

- 执行 RAG 检索的轮次，保存用户消息时传入 `embedInRag=false`
- 提醒分支保存用户消息时传入 `embedInRag=true`
- 目的：避免「找图/搜记录」类查询文本污染向量集合

### 5.4 AI 能力矩阵（7 大能力）

| 能力 | 触发方式 | 实现细节 |
|------|----------|----------|
| **智能对话** | 所有文本输入 | 基于 RAG 的上下文感知对话，SSE 流式输出，multi-turn 历史上下文 |

## 六、PDF 处理技术详解

### 6.1 PDF 文本抽取（三条路径）

| 路径 | 场景 | 技术 | 执行位置 |
|------|------|------|----------|
| 用户文件记录 | 上传 PDF 作为记录 | PDFBox 3.0.2 + LayoutPreservingTextStripper | Java 异步线程 |

#### LayoutPreservingTextStripper

自定义 `PDFTextStripper` 子类，保留 PDF 中表格的布局结构：
- 优先使用布局保留模式
- 失败时自动回退到纯文本 `PDFTextStripper`
- 对扫描件：检测文本长度 < 50 字时，提取内嵌图片 → 上传 OSS → 用于图像理解

### 6.2 PDF 纪念册生成

- **结语页**：AI 基于记录内容生成温暖寄语，失败时使用默认文案
- **字体**：支持中文（从 classpath 加载）
- **图片处理**：HTTP 下载远程 OSS 图片后嵌入 PDF

## 八、健康分析技术

### 8.1 体重管理

- **参考数据**：`WeightReference.java` 内置孕周体重增长参考范围
- **分析流程**：
  1. 获取用户首次体重记录作为基线
  2. 计算相对增重 = 当前体重 - 首次体重
  3. 查参考范围，判定状态（below/within/above）
  4. `@Async` 调用 AI 生成 2-4 句建议

### 8.2 胎儿 B 超分析

- **参考数据**：`FetalReference.java` 内置 12-40 周完整对照表（BPD/HC/AC/FL/EFW）
- **分析流程**：
  1. 将参考对照表 + 用户 B 超数据拼成 prompt
  2. 使用 `FETAL_SYSTEM_PROMPT` 约束 AI 角色（孕期B超解读顾问）
  3. 生成 2-3 句简洁建议
  4. 支持同步（带超时）和 `@Async` 异步两种调用方式
  5. 同步超时时自动转为异步

### 8.3 情绪-孕周聚合分析

- **数据来源**：合并 memo（心情字段）+ user_daily_log（每日心情）+ mood_record（多次心情记录）
- **按孕周聚合**：通过末次月经日计算孕周索引，按周汇总情绪分布与体重变化
- **情绪分类**：正向（happy, calm, peaceful）/ 负向（tired, anxious）
- **趋势计算**：`computeTrend()` 判断 stable/fluctuating/need_support
- **配偶关怀联动**：趋势为 fluctuating/need_support 时触发配偶关怀提醒

## 十二、关键技术创新点总结

1. **多源异构分块策略** — 长文档（知识库）在 RAG 侧统一分块（1500/100 + 中文分隔符优先）；短文档（记录/消息/产检/图片描述）整条嵌入。兼顾检索精度与语义完整性
2. **PDF 双路径处理** — Java 侧 PDFBox（含自定义 LayoutPreservingTextStripper 保留表格布局）抽取用户文档；Python 侧 PyPDFLoader 处理知识库分块。各司其职
3. **向量库与业务库协同** — MySQL 管权限与一致性，Milvus 管语义检索；通过 user_id 实现多租户隔离，source/source_id 实现可追溯
4. **Redis 容错降级** — 自定义 CacheErrorHandler 静默处理缓存异常，Redis 故障时自动降级到 MySQL 查库，业务零中断
5. **SSE 全链路流式** — 大模型 → Spring Boot → Next.js API Route (undici) → 浏览器，端到端流式传输；前端 AbortController 2 分钟超时保护
6. **检索轮次不嵌入** — 执行 RAG 检索的对话轮次不将用户问题写入向量库，避免「找图/搜记录」类查询污染检索集合
7. **双层提示词管理** — 数据库优先 + JSON 兜底，支持运行时热更新与模板变量替换
8. **异步嵌入** — `@Async` 异步执行 PDF 抽取、向量嵌入、健康分析，接口响应不受影响
9. **关系图谱雏形** — MySQL 外键关系构成用户-家庭-成员-记录的图结构，预留 Neo4j 扩展空间
10. **情绪-孕周智能联动** — 合并多数据源（memo + daily_log + mood_record）按孕周聚合，趋势分析联动配偶关怀定时推送
11. **定时提醒全链路** — 自然语言 → 多轮收集（最多2轮）→ 时间解析（含 LLM 兜底）→ 入库校验 → 到期 LLM 改写 → 站内通知 + 邮件
12. **健康分析双模式** — B 超分析支持同步（带超时自动转异步）和纯异步两种模式，内置完整的 12-40 周胎儿参考对照表
