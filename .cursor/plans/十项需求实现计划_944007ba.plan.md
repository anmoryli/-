## 2. 孕期时长放宽 3 个月

**理解**：当前多处将孕周限制在 4～40 周；放宽 3 个月 ≈ 约 12～13 周，即上限由 40 改为 **52 或 53**（取 52 周为 1 年，便于理解）。

**需改位置**（均为“上限”或“范围”判断）：

- [backend/.../AiController.java](backend/src/main/java/com/anmory/yunji/controller/AiController.java)：`weeklyTip`、`weeklyReminder`、`babyGrowth` 等处 `Math.min(40, ...)` → `Math.min(52, ...)`，`if (w > 40)` → `if (w > 52)`。
- [backend/.../PregnancyWeekUtil.java](backend/src/main/java/com/anmory/yunji/utils/PregnancyWeekUtil.java)：若有“40+周”或上限 40 的返回/判断，改为 52。
- [backend/.../EmotionPregnancyServiceImpl.java](backend/src/main/java/com/anmory/yunji/service/impl/EmotionPregnancyServiceImpl.java)：按孕周聚合时 `weekIdx > 40` 的过滤改为 `> 52`。
- [backend/.../EmotionPregnancyDailyHintServiceImpl.java](backend/src/main/java/com/anmory/yunji/service/impl/EmotionPregnancyDailyHintServiceImpl.java)：`week > 40` → `week > 52`。
- [backend/.../ProactiveIceBreakServiceImpl.java](backend/src/main/java/com/anmory/yunji/service/impl/ProactiveIceBreakServiceImpl.java)：`Math.min(40, ...)` → `Math.min(52, ...)`。
- 前端 [frontend/lib/pregnancy.ts](frontend/lib/pregnancy.ts) 等若有孕周上限常量或展示“40+”，同步改为 52。

**建议**：在 `PregnancyWeekUtil` 或常量类中定义 `MAX_PREGNANCY_WEEK = 52`，后端统一引用；前端若有对应常量一并改为 52。

## 3. PDF 导出 AI 总结退避

**目标**：AI 总结失败或导出过程异常时，不中断导出，使用默认结语，保证“导出一定成功”。

**现状**：[PdfExportServiceImpl.generateAiSummary](backend/src/main/java/com/anmory/yunji/service/impl/PdfExportServiceImpl.java) 已 catch 异常并返回 null，[addAiSummary](backend/src/main/java/com/anmory/yunji/service/impl/PdfExportServiceImpl.java) 在 null/empty 时已有默认句。退避需加强两点：

1. **调用处兜底**：在 `exportToPdfInternal` 中调用 `addAiSummary` 时用 try-catch 包裹；若抛出任何异常（如 DocumentException、字体等），捕获后不再调用 `addAiSummary`，改为向 PDF 追加一页仅含**固定默认结语**（与当前默认句一致或稍通用，如“愿这段珍贵的孕期时光，成为您与家人最温暖的回忆。祝宝宝健康出生，家庭幸福美满。”），并打日志。
2. **默认句统一**：将默认结语提成常量或私有方法，`generateAiSummary` 返回 null 时与上述异常兜底共用同一段文案。

这样无论 AI 超时、网络错误还是写 PDF 异常，导出都能完成并有一页结语。

## 4. 邮箱支持切换绑定

**目标**：用户已绑定邮箱 A 时，可改为绑定邮箱 B（换绑）；校验新邮箱未被他人占用，当前用户解绑旧邮箱再绑定新邮箱。

**后端**：

- [UserServiceImpl.bindEmail](backend/src/main/java/com/anmory/yunji/service/impl/UserServiceImpl.java)：  
  - 若传入的 `email` 与当前用户已有 email 相同，可视为“重复绑定”直接成功或 no-op。  
  - 若不同：先按现有逻辑校验“该邮箱是否已被**其他**用户绑定”（`selectByEmail` 且 `userId` 不等于当前用户）；若被他人占用则抛 BusinessException。  
  - 然后更新当前用户：`user.setEmail(email)` 并保存（即允许从 A 换到 B）。
- 无需单独“解绑”接口：换绑即覆盖；若产品需要“仅解绑不绑新邮箱”，再单独加一个解绑接口（当前需求仅“支持切换绑定”）。

**前端**：

- [frontend/app/(app)/profile/bind-email/page.tsx](frontend/app/(app)/profile/bind-email/page.tsx)：当 `user.email` 已存在时，当前仅展示“已绑定邮箱”。改为：保留展示当前邮箱，并增加“更换邮箱”入口（如按钮），点击后展示输入框让用户输入新邮箱，提交时调用现有 `bindEmail(userId, newEmail)`；成功后 `setUser({ ...user, email: newEmail })` 并 toast“邮箱已更换”或“绑定已更新”。
- 文案：已绑定状态下可写“更换邮箱”“绑定其他邮箱”等温和表述。

## 5. 全屏渐变背景 + 大操作时加速动效

**目标**：

- 整页背景为**缓慢渐变**，约 3 色（当前背景色 + 两种低饱和度、与当前风格相近的颜色），渐变有明显但舒缓的动画。
- “大操作”时（如发布笔记成功、切换底部 Tab/路由）渐变动画**短暂加速**，再恢复为慢速。

**实现要点**：

- **颜色**：在 [frontend/app/globals.css](frontend/app/globals.css) 的 `:root` 中增加 2 个变量，如 `--gradient-b`、`--gradient-c`（低饱和度、与 `--background` 同色系），或直接使用现有 `--accent-1-muted`、`--accent-3-muted` 的浅色。
- **布局**：在 [frontend/app/(app)/layout.tsx](frontend/app/(app)/layout.tsx) 中，将最外层容器设为全屏渐变背景（例如 `background: linear-gradient(...)` 使用上述 3 色），并留出内容区（如 `max-w-lg`、`min-h-screen` 的 main）。
- **缓动动画**：用 CSS `@keyframes` 实现渐变位置/角度的缓慢变化（例如 20s～30s 一周期），默认应用在该背景层。
- **加速触发**：  
  - 方式 A：用 React context 或全局状态存“是否处于加速中”；在“发布笔记成功”的回调、以及路由变化（如 `usePathname()` 变化）时，置 true，持续 1.5～2s 后置 false。  
  - 方式 B：在相同时机给 body 或根节点加一个 data 属性/class（如 `data-motion-fast`），CSS 中当存在该 class 时使用另一段更快的 `@keyframes`（如 3～5s 周期）。
- **“大操作”范围**：按你的要求仅限：发布笔记成功、切换底部 Tab（路由变化）；普通点击、输入不触发。

## 6. 孕期记录统计：总 / 妈妈 / 爸爸

**现状**：[FamilyServiceImpl.getCreatorPregnancyInfo](backend/src/main/java/com/anmory/yunji/service/impl/FamilyServiceImpl.java) 仅返回 `recordCount` = 创建者（妈妈）的 memo 数；[profile/page.tsx](frontend/app/(app)/profile/page.tsx) 对家庭成员展示为“孕期记录：creatorPregnancy.recordCount”。

**目标**：展示三项——**总条数**、**妈妈的**、**爸爸的**（爸爸指家庭内配偶，若有多成员仅统计“配偶”角色对应的 userId 的 record 数）。

**后端**：

- `getCreatorPregnancyInfo` 增加字段（或新接口，避免破坏现有结构）：  
  - `recordCountTotal`：该家庭 creator + 所有配偶成员的 memo 条数之和（或按“可见范围”能看到的记录总数，若当前 API 是“家庭成员看创建者”的统计，则 total = creator 数 + 配偶数）。  
  - `recordCountMom`：creator 的 count。  
  - `recordCountDad`：配偶（isSpouse=true）的 userId 的 memo count 之和。
- 需要 [MemoMapper](backend/src/main/java/com/anmory/yunji/mapper/MemoMapper.java) 已有 `countByUserId`，对家庭内 creator + 配偶分别 count 再汇总即可。

**前端**：

- [frontend/app/(app)/profile/page.tsx](frontend/app/(app)/profile/page.tsx)：当为家庭成员时，若新接口返回了 `recordCountTotal`、`recordCountMom`、`recordCountDad`，在“孕期记录”区域展示为三块或一行内“总 / 妈妈 / 爸爸”（如“共 12 条 · 妈妈 8 · 爸爸 4”）。孕妇本人视角若也需要“总/妈/爸”（例如自己+配偶），需后端为孕妇也提供对应统计（如 getMyFamily 下 creator+spouse 的 count），前端同风格展示。

## 7. 家人共享「生成本周任务」：AI + RAG + 本周去重

**现状**：[FamilyTaskServiceImpl.generateTasksForWeek](backend/src/main/java/com/anmory/yunji/service/impl/FamilyTaskServiceImpl.java) 使用**写死**的两条任务（“本周陪同产检”“每日一个拥抱”），且无“本周已发送”限制。

**目标**：

- 生成本周任务改为：**RAG 检索**（近期记录/孕周等） + **AI 生成**（与 [suggestTasksForWeek](backend/src/main/java/com/anmory/yunji/service/impl/FamilyTaskServiceImpl.java) 类似的 prompt），生成若干条任务并落库、推送给配偶（站内+邮件）。
- 若**当前孕周已为本家庭生成并推送过**，则本次请求返回明确提示：“本周已发送过，如需继续添加请到爸爸成长营添加”，不再生成。

**实现要点**：

- **去重**：在 DB 或 Redis 记录“某 familyId 在某孕周已执行过 generateForWeek”。例如表 `family_task_week_sent(family_id, pregnancy_week, created_at)` 或 Redis key `family:task_sent:{familyId}:{week}`。在 `generateTasksForWeek` 开头：取当前 creator 的孕周，查是否已存在该 familyId+week 记录；若存在则抛 BusinessException（或返回 409）并提示上述文案。
- **生成逻辑**：复用或调用与 `suggestTasksForWeek` 相同的 AI+Prompt 逻辑（可抽成私有方法），得到 `List<Map<String,String>>`（title/description），然后对每条调用 `createTask(familyId, assigneeUserId, title, description, taskType)`；createTask 内部已有站内通知 + 邮件给配偶，无需重复。
- **RAG**：在生成前用 [RagService](backend/src/main/java/com/anmory/yunji/common/RagService.java) 对 creator 的 userId 做一次检索（如“近期记录 孕周 产检 陪伴”），将检索摘要拼入 AI 的 prompt（例如在 pregnancy.json 的 `family_task_suggest` 或新 key 中增加“近期摘要”占位符），使生成的任务更贴近期情况。
- **前端**：[family/page.tsx](frontend/app/(app)/family/page.tsx) 的“生成本周任务”按钮，请求 `generateTasksForWeek`；若后端返回 409 或明确错误码/文案，toast 显示“本周已发送过，如需继续添加请到爸爸成长营添加”，并可选高亮“爸爸成长营”入口。

## 8. 爸爸完成爸爸训练营任务 → 通知妈妈

**目标**：爸爸完成一条任务后，向**妈妈**（家庭创建者）发送站内通知 + 邮件（若妈妈绑定了邮箱）。

**实现**：

- 在 [FamilyTaskServiceImpl.completeTask](backend/src/main/java/com/anmory/yunji/service/impl/FamilyTaskServiceImpl.java) 中，`complete` 成功后：  
  - 根据 `familyTask` 的 `familyId` 取家庭信息，得到 `creatorUserId`（妈妈）。  
  - 调用 [UserNotificationService.notifySystem](backend/src/main/java/com/anmory/yunji/service/UserNotificationService.java)(creatorUserId, title, body)，title 如“爸爸完成了一条小任务”，body 可含任务标题或“快去夸夸他吧”等简短文案。  
  - 若妈妈有邮箱，调用 [MailService](backend/src/main/java/com/anmory/yunji/service/MailService.java) 发一封简短邮件（如“老公完成了「xxx」任务，快去夸夸他吧”），与 app 风格一致（见下条）。
- 避免重复：仅当本次操作将任务从 pending 变为 completed 时发送（当前代码已判断 `!"completed".equals(t.getStatus())` 才执行 complete，所以一次完成只触发一次）。

## 9. 文件/语音上传缺少 userId 导致 400

**现象**：上传文件记录或录音时出现 “Required request parameter 'userId' for method parameter type Integer is not present”；上传照片正常。

**可能原因**：multipart 请求在网关/代理或 Spring 解析时，部分环境下非文件字段未正确绑定到 `@RequestParam("userId")`。

**方案**：

- **前端**：在 [frontend/lib/api/memo.ts](frontend/lib/api/memo.ts) 的 `addVoice`、`addFile` 调用处，将 `userId` 同时放在 **URL 查询参数** 中，例如 `apiUpload(..., "/api/memo/addVoice?userId=" + userId, ...)` 或封装为 `apiUpload(..., path + (path.includes('?') ? '&' : '?') + 'userId=' + userId, ...)`，确保后端一定能从 query 拿到 userId。
- **后端**：在 [MemoController.addVoice](backend/src/main/java/com/anmory/yunji/controller/MemoController.java)、[addFile](backend/src/main/java/com/anmory/yunji/controller/MemoController.java) 中，将 `@RequestParam("userId")` 改为 `@RequestParam(value = "userId", required = false) Integer userId`，若为 null 则从 query 再取一次（Spring 允许同一参数既在 query 又在 body）；若仍为 null 返回 400 并提示“缺少 userId”。这样无论前端用 query 还是 form 传，都能兼容。
- 若项目有 Next.js 或 Nginx 代理：确认 multipart 请求被正确转发且不丢弃 query 或 form 字段。

## 10. 邮件 HTML 与 app 一致：配色 + 渐变 + 缓动

**目标**：所有发送的 HTML 邮件（任务提醒、配偶提及、爸爸完成任务通知、导出 PDF 等）使用与 app 一致的配色和**缓慢渐变背景 + 缓动动画**。

**实现**：

- 在 [MailServiceImpl](backend/src/main/java/com/anmory/yunji/service/impl/MailServiceImpl.java) 中扩展或新增“带渐变与动画”的 HTML 模板方法（如 `wrapHtmlBodyWithStyle`）。  
  - 使用与 [globals.css](frontend/app/globals.css) 一致的色值：如背景 `#FDFAF7`、强调色 `#E3B8B0`、`#B8CBB3` 等，用内联 style 写死（邮件客户端对 CSS 支持有限）。  
  - 背景：`background` 使用 `linear-gradient` 三色（与需求 5 中选定的 3 色一致），并配合 `animation` 做缓慢的渐变移动（例如 `background-position` 或 `background-size` 的 keyframes，周期 15～25s）。  
  - 注意：部分邮件客户端不支持或会禁用 CSS 动画，需保留无动画时的渐变背景作为 fallback，且核心内容不依赖动画可读。
- 将当前使用 [wrapHtmlBody](backend/src/main/java/com/anmory/yunji/service/impl/MailServiceImpl.java) 或纯文本的地方，改为使用新模板；任务提醒、配偶提及、爸爸完成任务、PDF 导出结果等邮件正文统一走该模板。
- 若某些邮件目前是 `sendTextMail`，可改为 `sendHtmlMail` 并传入上述 HTML 片段，保证视觉统一。
